The Rise of NoSQL

Posted on 2015-09-20   |   In Programming

Below, I’m going to attempt to sum up some of the points Martin Fowler made in his Introduction to NoSQL talk at the GOTO Aarhus Conference 2012, mostly to hammer these concepts into my own forgetful brain. Fowler, who started developing software in the 80s, has written several notable books on topics including, but not limited to refactoring, application architecture, and NoSQL. Here is a link to the talk.

Problems with SQL

One of the central problems with relational databases is the “Object-Relational Impedance Mismatch.” Simply put, there is a gap between object-oritented design principles and databases. It’s difficult to map objects to databases. We can have a logical, cohesive user interface that makes perfect sense to the human eye, but when we try and store all this data, we are forced to split it up into a bunch of different tables linked together by relations.

Wouldn’t it be so much simpler to take the object and just drop it directly into the database? In the mid-90s, some smart people attempted to push the idea of Object Oriented Databases that would do just this. Unfortunately, the databases in place were overwhelmingly relational and the integration process ultimately prevented the OO databases from ever gaining significant traction. So instead of a complete paradigm shift, developers formed abstraction layers between relational databases and OO code bases called Object Relational Models. With these, large corporations that had invested heavily in storing their data in a relational way did not have to pay the high costs of migration from one database type to another. Developers could code their OO apps and companies didn’t have to mess with their data stores. Everyone was happy-ish.

What Changed?

As the early 2000s rolled around and the internet rose to prominence, software engineers faced a new set of challenges. Thousands of people were accessing websites simultaneously and developers scrambled to allow levels of scalability within their websites that could handle the traffic. One option was to increase their server size. Just buy a bigger box. SQL was designed for this. But this was expensive and provided provided room for a relatively small number of failure points, so companies like Google and Amazon began experimenting with alternate solutions. How about buying a bunch of little boxes instead and spreading all this data out? Fowler mentioned that engineers whom he spoke with who had tried making relational databases work across this type of distributed system often described the process using the phrase: “unnatural acts.” Ultimately, this led to the advent of the NoSQL databases.

NoSQL is a term that was coined as a Twitter hashtag for an early conference focused on such languages. It eventually just stuck. Fowler describes a set of common characteristics of NoSQL databases:

  • Non-relational
  • Open Source
  • Cluster-friendly
  • Works with 21st century website environments
  • No Set Schema

Data Model

When he says “no set schema,” he clarifies that there still ends up being an implicit schema when queries are made. If a query is made for an order, it may expect an order number and price. This implies a schema exists. This implicit schema still needs to be managed.

Mongodb.com defines these as “dynamic schema.” In an agile development environment, it’s common to have to alter database schemas frequently as new features are added. “There’s also no way, using a relational database, to effectively address data that’s completely unstructured or unknown in advance.”

Benefits

Relational databases naturally scale vertically. They require a single server to grow as the data volume grows. NoSQL DBs are designed to scale horizontally, often by supporting auto-sharding, load balancing, and/or automatic replication. Data can be spread among many cheap servers without the application needing to be aware of what’s going on behind the scenes.

Aggregate Oriented

The aggregate in Fowler’s words is often the highest-level “object” we are manipulating. In a document-focused DB like MongoDB, the aggregate would be the document. It is a single unit that we move back and forth from database to application. There is a type of NoSQL database that is not aggregate-oriented (graph) but I’m not going to get into that here.

NoSQL and Consistency

Consistency is the idea that when multiple people are accessing and making changes to a database, the data will remain consistent. Customer A should not be able to make a change to Product A while Customer B makes a different change at the same time, effectually overwriting Customer A’s. NoSQL databases are arguably weaker in this area compared to their relational counterparts. They have often been described as “eventually consistent.”

Fowler talks about an “offline lock” system to be used in such cases. Each data record or aggregate gets a version stamp. When a user retrieves this record, they also receive the stamp. If the user manipulates it and attempts to post back to the db and there is a version mismatch, they’ll receive an error. The version mismatch would be caused by another user updating the same record prior.

This type of consistency is “logical consistency.” It applies whether the data is on one server or distributed across many. Sharding data, or taking a copy of the data and splitting into shards on several machines brings up logical consistency problems.

Replicating data, or copying the same piece of data across multiple machines brings up different problems. It’s advantageous in terms of performance, i.e. more nodes handling the same request and resilience, i.e. if one node goes down, other nodes keep going. But, what happens if two people are making a request on a distributed DB and the communication line between the replicated DBs goes down? We can deny both requests and tell the users, “Sorry, there’s been an error with the network.” In this case, we ensure we aren’t going to run into data consistency issues but are giving the user a bad experience. Or, we can accept both requests and leave it up to the business to deal with it later down the line. For example, two people book the same room because of this and the hotel eventually just moves one of them into an alternate room. So essentially it’s a choice between consistency (1st option) and availability/response time, the latter of which says “Yes, I’ll keep going at the risk of introducing inconsistent data.” The more consistency you want, the more nodes have to communicate which can lead to a more expensive and slower operation. Fowler argues that this decision should ultimately be left up to the business.

When and why you’d want to use NoSQL

  • You’ve got to deal with large amounts of data.
  • You want to develop quickly.
  • You need to deal with analytics, i.e. pulling data from a wide variety of sources (excel, log files) with different structures.

Is NoSQL the future?

Fowler believes the future lies in “polyglot persistence.” He thinks there will be room for many different types of DBs and it will be up to the developer to determine which is appropriate for the problem at hand.

Big O and Time Complexity

Posted on 2015-09-16   |   In Programming

O(f(x)) is a mathematical term. “O” means order, as in “order of complexity” and big means… big (“O” not “o”.)

Big O notation allows us to analyze algorithms in an abstract fashion. It falls under the umbrella of complexity analysis. Most programming languages have a set of tools (such as profilers) that can measure the time it takes to run a particular chunk of code down to the millisecond. Big O is, by definition, a measure of our algorithm’s total number of machine calculations as we increase the input value towards infinity. It excludes coefficients and lower order terms. It is not a way to determine the exact time a process will take to complete in a certain language, but rather a higher level measurement of how complex an algorithm is.

Simply put:

Complexity analysis explains how an algorithm will behave as it’s input grows

We’re interested in our algorithm’s behavior as n tends towards infinity. The way we analyze this in computer science differs from how one would formally analyze O(f(x)) in mathematics. In mathematics we would include coefficients, constants, and lower order terms. For example:

Is filtered down in comp sci Big O notation to

f(n) = n

The idea behind this is that we want to isolate the parts of the function that will be effected by the growth of n. If we change n, does the constant 408 change? No. If we change n, does 10n behave differently than 20n would? No, not really. 20n would increase at the same linear rate as 10n, so overall, it’s behavior is the same. So we filter it down to a very basic level and say the function will change at a linear rate: f(n) = n which is expressed in Big O notation as O(n)

Another filter that is applied here by computer scientists and not mathematicians is to drop the terms that grow slowly and keep those that grow quickly as n increases. If we have a function f(n) = n<sup>2</sup> + 10n + 408, we will drop the constant and remove the multiplier 10 as we did before. But, we will also remove n from the leftover f(n) = n<sup>2</sup> + n since n will grow at a negligible rate compared to n2. What we are left with is f(n) = n<sup>2</sup>.

Some examples of common Big O time complexity are:

O(1) - Constant Time

Total number of calculations stays the same no matter how large or small our input is

O(n) - Linear Time
Total number of calculations grows linearly in relation to the input

O(n2) - Quadratic Time
Time execution is proportional to the square of the input size
